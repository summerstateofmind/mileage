# Mileage V0.3 — Design Spec

**Date:** 2026-05-23
**Status:** Draft
**Goal:** Replace the known-broken V0.1.1 YPT log-penalty formula with an academically defensible **log-normal CDF over composite outcomes**, ship proper Code Survival at 30 days, and lay the foundation for opt-in Quality Mode. After V0.3, YPT can come above the fold for the Phase 2 standards-play asset; until then it's still below the fold but it is no longer producing negative numbers as a normal outcome.

This spec covers V0.3 in three shippable phases:

- **Phase 1 — YPT v2 math.** Log-normal CDF over yield_rate, Bayesian shrinkage during cold start, stratification by model, validity predicate, `mileage explain ypt` upgraded with formula + literature anchors + version stamp.
- **Phase 2 — Code Survival (proper).** Add 30-day survival window alongside existing 7-day; background recompute on sync.
- **Phase 3 — Quality Mode (opt-in, crosses the diff-read line).** `mileage enable quality-mode`, cognitive complexity delta, jscpd duplication, Semgrep diff-aware scanning, LLM-as-judge intent fulfillment (local Qwen via Ollama).

**Phase 1+2 ship together in this version.** Phase 3 is sketched here for design coherence but gets its own implementation session because the dependency surface (tree-sitter, lizard, jscpd, semgrep, Ollama + ~5GB model) and the privacy-line-crossing nature deserve dedicated treatment.

---

## Critical scope decisions

### What we ARE building (Phase 1 + 2)

- **YPT v2 (log-normal CDF over composite yield_rate).** Replaces V0.1.1 entirely. Stamped `YPT-2026.1`.
- **Composite outcome counting.** Survival-weighted attribution + self-tag overlay. (LLM-as-judge is Phase 3.)
- **Bayesian shrinkage from literature prior** during cold start (low N).
- **Stratification by model.** Per-model yield_rate, then weighted-average overall.
- **Validity predicate.** Sessions/days failing the threshold report "not scored" instead of YPT 0 — the Strava pattern.
- **Code Survival 30d.** Existing 7d evaluator already handles parameterized window_days; we just call it for 30 too and surface both numbers.
- **Background recompute.** Every `mileage sync` re-evaluates survival for commits that are old enough but unevaluated for their window.
- **Calibration storage.** A small `calibration` table holding the current `(μ, σ)` and the recipe used to derive them. Lets us recompute snapshots when calibration shifts without redefining the schema.

### What we ARE NOT building (defers)

- **Phase 3 Quality Mode.** Tree-sitter + lizard + jscpd + Semgrep + Ollama Qwen-judge. Own session. The opt-in machinery, the explicit data-read disclosure printout, and the heavy installs all need separate care.
- **LLM-as-judge intent fulfillment.** Lives in Phase 3 because it requires a local Ollama server.
- **Annual recalibration / concordance tables.** That's V2.0+ standards-play work. V0.3 ships hardcoded v0 calibration with explicit "expect change" labeling.
- **IRT (Item Response Theory) task difficulty.** Research-validated long-game; requires tagged-task corpus we won't have for years.
- **Per-language stratification.** V0.4 backlog (`Per-language Cost-per-Ship and YPT breakdown`).
- **Cross-tool stratification.** V0.4 (Cursor, Copilot ingestion).

### What changes for users

| | Before V0.3 (V0.2 / V0.1.1) | After V0.3 (Phase 1+2) |
|---|---|---|
| YPT formula | `(10·d + 5·i) − 5·log₁₀(tokens/100K)` (negatives are normal) | `100 × Φ((ln yield_rate − ln μ)/σ)` (bounded 0–100) |
| YPT version stamp | `v0.1.1` | `YPT-2026.1` |
| Sub-threshold days | YPT shown as negative number or near-zero | YPT shown as "not scored" |
| Per-model breakdown | Not surfaced | Stored in `provenance.by_model`; visible in `mileage explain ypt` |
| Code Survival | 7d only | 7d + 30d, both surfaced when data is mature enough |
| `mileage explain ypt` | Shows v0.1.1 formula + "this is broken" caveat | Shows YPT-2026.1 formula, calibration, per-model breakdown, citations |

The headline `mileage show` UI does not change drastically — YPT remains below the fold; the goal is honest numbers, not new UI surface.

## Phase 1 — YPT v2 math

### Formula

```
yield_rate_per_session  = composite_outcomes / (tokens / 100,000)
score(yield_rate)       = 100 × Φ((ln yield_rate − ln μ) / σ)
```

Where:
- `Φ` is the standard normal CDF.
- `(μ, σ)` are anchored to two empirical control points (Lighthouse pattern).
- `composite_outcomes` per session = Σ(attribution_weight × survival_weight) over commits attributed to the session, possibly adjusted by self-tag.

**Initial calibration (v0, explicitly labeled "expect change"):**

- Anchor: P50 yield_rate → score 50; P10 yield_rate → score 90.
- Initial prior values: `μ₀ = ln(0.5) ≈ -0.693` ("median user ships 0.5 outcomes per 100K tokens"), `σ₀ ≈ 1.26` (so P10 yield_rate ≈ 5× median maps to z = 1.28).
- These will be wrong; that's the point of two-point anchoring + Bayesian shrinkage. They get less wrong as data accumulates.

**Attribution weights (per attributed commit):**

| Tier | Weight |
|---|---|
| `direct` (tool call to `git commit`) | 1.0 |
| `high` (post-commit hook, V0.2.b2+) | 0.85 |
| `inferred` (proximity heuristic) | 0.5 |

**Survival weight (per attributed commit):**

| Condition | Weight |
|---|---|
| Survival not yet evaluated (commit younger than window) | 1.0 (treat as surviving until proven otherwise) |
| Survival evaluated, file-level proxy | `max(0.1, lines_surviving / lines_added)` (floor at 0.1 so a fully reverted commit still counts a bit) |
| Survival evaluated, lines_added = 0 (refactor with no net new lines) | 1.0 |

**Self-tag overlay:**

| Tag | Effect on composite_outcomes |
|---|---|
| `shipped` | Adds a synthetic outcome of weight 1.0 if no attribution exists for the session (user vouched it shipped even if we didn't catch the commit) |
| `dead-end` | Zeroes any attributed outcome weight for the session (user vouched it didn't work even if a commit landed) |
| `exploring` | Excludes session from the day's denominator entirely (research ≠ waste) |
| `debugging` | No effect on weights — informational |
| (no tag) | Defaults: attribution drives outcome, no overlay |

### Bayesian shrinkage (cold start)

Before users have enough data, anchor the score to the literature prior with weight that shrinks as N grows:

```
N_0           = 100   (pseudo-sample size; reflects confidence in the prior)
μ_empirical   = median(ln yield_rate) over recent sessions (winsorized at P1/P99, MAD-based)
σ_empirical   = 1.4826 × MAD(ln yield_rate)   (MAD-to-σ for log-normal)
μ_effective   = (N_0 × μ_0 + N × μ_empirical) / (N_0 + N)
σ_effective   = (N_0 × σ_0 + N × σ_empirical) / (N_0 + N)
```

For a brand-new install: N=0 → μ_effective = μ_0, σ_effective = σ_0 (pure prior).
After 100 sessions: 50/50 weight.
After 10,000 sessions: essentially empirical.

Provenance records both the prior and the empirical estimate; `mileage explain ypt` shows both and which dominates.

### Stratification by model

Compute yield_rate per `model_id` over the rolling window, then either:
1. **Per-model YPT** (display): `score_model = score(yield_rate_model)` shown in `mileage explain ypt`.
2. **Overall YPT** (snapshot field): session-count-weighted average of per-model scores.

Stratification stored in `snapshot.provenance.by_model`:
```json
"by_model": {
  "claude-opus-4-7": { "sessions": 47, "outcomes": 18.8, "yield_rate": 0.40, "score": 42 },
  "claude-sonnet-4-6": { "sessions": 23, "outcomes": 16.0, "yield_rate": 1.74, "score": 78 }
}
```

### Validity predicate

A session is **scorable** if any of:
- It has ≥1 attributed commit (direct/high/inferred), OR
- It carries a self-tag of `shipped`, `debugging`, or `exploring`.

A session is **not scorable** if:
- It has 0 attributions AND
- It has no self-tag OR is tagged `dead-end` with 0 attributions (already covered — a tagged dead-end implies work happened, but yields nothing measurable).

A snapshot day is **scorable** if it has ≥1 scorable session. Days with only unscorable sessions report `ypt_score = null` (rendered as "not scored — explore your `mileage tag` queue").

This is the Strava pattern (no GPS → "activity not scored"). Exploration is real work; we don't punish it with a 0.

### Calibration storage

New table holds the active `(μ, σ)` and is the single source of truth for snapshot recomputation:

```sql
CREATE TABLE IF NOT EXISTS calibration (
  version       TEXT PRIMARY KEY,         -- e.g. 'YPT-2026.1'
  mu            REAL NOT NULL,
  sigma         REAL NOT NULL,
  n_prior       INTEGER NOT NULL,         -- pseudo-N for shrinkage
  anchor        TEXT NOT NULL,            -- 'P50→50, P10→90'
  source        TEXT NOT NULL,            -- 'v0 literature prior' | 'empirical N=...'
  created_at    INTEGER NOT NULL,
  active        INTEGER NOT NULL DEFAULT 0
);
```

`mileage sync` seeds the v0 row if no calibration exists. Future recalibration (V2.0+) writes a new row and flips `active`.

## Phase 2 — Code Survival (proper)

The existing `src/compute/survival.ts` already takes `windowDays` as a parameter and `commit_survival` already has `window_days` in its PK. The work is mechanical:

1. **Call the survival evaluator for both windows.** `updateSurvivalForCwd(db, cwd, 7)` and `updateSurvivalForCwd(db, cwd, 30)` on each `mileage sync`. The selector skips commits that already have a row for that window.
2. **Background catch-up.** Same loop catches up any commit that's now old enough to evaluate.
3. **Per-window summaries.** `getSurvivalSummarySince` already takes `windowDays`; render layer calls it for both 7 and 30.
4. **Use 30d survival as the canonical survival_weight in YPT.** When both are available, 30d is the more honest signal. When only 7d is available (commit is 7–30 days old), use 7d as a fallback.

### Render-layer change

`renderSurvivalBlock` shows both numbers when both have ≥1 commit evaluated:

```
  Code health       91% alive at 7d  ·  84% alive at 30d   (12 / 9 commits evaluated)
```

If only one window has data:

```
  Code health       87% alive at 7d   (12 commits evaluated)
```

## Phase 3 — Quality Mode (sketched; own session)

> **This phase is not implemented in V0.3 Phase 1+2.** It's sketched here so the design coherence is captured. When scheduled, it gets its own spec + plan.

### Opt-in machinery

`mileage enable quality-mode` is a discrete CLI verb. Behavior:

1. Print exactly what data Quality Mode reads from the working tree, with file paths and the specific commands it will run.
2. Print exactly what data leaves the machine: **nothing** (Ollama runs local; static analyzers are local).
3. Require typed confirmation (`yes I read this`) before flipping the bit.
4. Write `quality_mode.enabled = true` to `~/.mileage/config.json`.
5. On uninstall (`mileage disable quality-mode`), purge any cached static-analysis outputs.

### Quality signals (each contributes to composite_outcomes when Quality Mode is on)

| Signal | Source | Weight in composite | Caveat |
|---|---|---|---|
| Compile/typecheck pass | `tsc --noEmit`, `mypy`, `pyright`, `cargo check` on touched files | 0.2 boost per clean diff | Skips on multi-language diffs to avoid false negatives |
| Cognitive complexity delta | tree-sitter + lizard | -0.1 to +0.1 depending on direction | Aggregate-only, never line-level |
| jscpd duplication delta | jscpd on diff vs repo | -0.1 if diff increases duplication | Skips for code under a "vendored" allowlist |
| Semgrep anti-patterns | Semgrep with conservative ruleset, diff-aware | -0.2 per new finding | User can disable individual rules |
| LLM-as-judge intent fulfillment | Local Ollama Qwen3-8B on `(commit_msg, diff)` | 0.0 to +0.3 | Confidence score, not ground truth (per CodeFuse-CommitEval) |

### Why Phase 3 needs its own session

- **Heavy external deps:** Ollama + Qwen3-8B (~5GB), Python (lizard, semgrep), Node (tree-sitter, jscpd). Some users won't want or have these.
- **First privacy-line crossing:** Quality Mode reads diff content. The opt-in flow has to be ironclad — clear, explicit, reversible, and audit-printable. That UX work is its own project.
- **LLM-as-judge calibration:** Bias mitigation (run twice with swapped order), confidence scoring, the "is the judge confident enough to count?" predicate — all need their own dogfooding loop.
- **Reversibility:** A wrong design here is hard to walk back. Better to ship Phase 1+2 first, dogfood the cleaner YPT, then layer Quality Mode on top with the lessons from a month of real use.

## User journey (Phase 1+2)

```
[1: User upgrades Mileage to V0.3, runs `mileage sync`]
  -> Schema migration creates `calibration` table
  -> Seed row written: YPT-2026.1, μ=-0.693, σ=1.26, source='v0 literature prior'
  -> Survival evaluator runs for windows [7, 30]
       -> Older commits that were never evaluated for 30d get evaluated now
  -> Snapshots are recomputed using YPT v2
       |
[2: User runs `mileage show`]
  -> Layout unchanged from V0.2 — YPT still below the fold
  -> YPT line shows a 0–100 number instead of a negative one
  -> Days that don't meet the validity threshold show "not scored"
  -> Code health line shows both 7d and 30d
       |
[3: User runs `mileage explain ypt`]
  -> Sees YPT-2026.1 formula
  -> Sees current calibration (μ, σ, anchor)
  -> Sees per-model breakdown (Opus 42, Sonnet 78)
  -> Sees citations
  -> Sees "v0 calibration; expect change" disclaimer
       |
[4: Author dogfoods for ~1-2 weeks, captures findings, refines for Phase 3]
```

## Edge cases

- **Empty DB / first sync:** N=0 → YPT uses pure prior. All days for a brand-new user will be "not scored" until the first attributed commit lands.
- **All sessions tagged `exploring`:** Day is "not scored" (denominator empty after exploration excluded). Render layer messages this gently.
- **Commit younger than 7 days:** Survival weight defaults to 1.0; snapshot will be recomputed once 7d evaluation completes.
- **Mixed-model day where one model has 1 session:** Stratification keeps it in `by_model` but the per-model score is high-variance — render layer shows the breakdown but the overall YPT averages cleanly.
- **MAD = 0 (all yield_rates identical, rare):** Fall back to `σ_effective = σ_0` from prior (don't divide by zero).
- **`lines_added = 0` commit (pure deletion or refactor):** Survival weight = 1.0 (no lines to track survival on; don't penalize).
- **Schema migration from V0.2:** Old snapshots in DB have V0.1.1 scores. After `mileage sync` runs the recomputer, all snapshots become YPT-2026.1. We do NOT preserve old scores — they were wrong; the version stamp in provenance is what matters.
- **User on V0.2 who never ran sync:** No data to recompute. First sync seeds calibration and computes V2 snapshots from scratch.
