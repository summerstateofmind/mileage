# Mileage V0.2 Phase 2 — Design Spec

**Date:** 2026-05-23
**Status:** Draft
**Goal:** Make Mileage equally compelling for **API users** (Phase 1's dollar wedge) and **Pro/Max subscribers** (Phase 2's cap-utilization + quality wedge). Add the forcing-functions that drive behavior change, not just data display.

This spec covers Phase 2 in three shippable sub-phases:
- **Phase 2a (the Max-plan wedge):** plan-aware display + tier-flex audit + rate-limit detection + stats-cache cross-check
- **Phase 2b (sticky behavioral features):** self-tag system + post-commit hook + behavioral patterns + Code Survival 7d
- **Phase 2c (visual polish + brand artifact):** heatmap + sparklines + bar charts + `mileage report --week` + `mileage review`

---

## Critical scope decisions

### What we are NOT building (and why)

- **`/api/oauth/usage` integration.** Anthropic's policy explicitly prohibits third-party use of subscription OAuth tokens. Endpoint also 429s aggressively into uselessness. See `docs/research/2026-05-23-oauth-policy-findings.md` (to be written).
- **Reading credentials file to auto-detect plan.** Conservative interpretation of the policy: don't touch `.credentials.json` at all. User declares plan via `mileage config set-plan`.
- **Anthropic Admin API integration.** Documented and clean, but only relevant for API-key customers with org-admin access. Push to V0.3.
- **Quality Mode** (diff-static-analysis, LLM-as-judge). Stays V0.3 — needs Quality Mode opt-in machinery.
- **YPT formula upgrade** (log-normal CDF). Stays V0.3.
- **Cross-tool integration** (Cursor, Copilot). Stays V0.4.

### What we ARE pulling forward from V0.3

- **Code Survival Rate (7d).** Git-only, no diff reading. Critical quality signal for Max-plan users who don't care about dollars.
- **Tier-flex audit.** Pure data join over what we already store (model + outcome). Highest-aha signal for the user's own Max-plan dogfooding.
- **Behavioral pattern detection.** Cheap to compute from existing data.

## Phase 2a — The Max-plan wedge

### User journey

```
[1: Max-plan user runs `mileage sync && mileage show`]
  -> If plan not configured: prompt
       "What plan are you on?
         [1] Claude Max ($100/mo)
         [2] Claude Max ($200/mo)
         [3] Claude Pro ($20/mo)
         [4] API pay-as-you-go
         [5] Cursor Pro ($20/mo)
         [6] Mixed / Other (use defaults)"
  -> Plan is saved to ~/.mileage/config.json
       |
[2: mileage show now displays Max-100 framing]
  -> Lead with "X% of your typical week" (self-baseline, never a fake cap%)
  -> Rate-limit hits prominent: "Rate-limited 0 times this week ✓" or "⚠ 3 times"
  -> Tier-flex audit when multi-model data present:
       "Your Opus sessions ship 0.4 commits/session; Sonnet ships 1.7.
        Are you getting Opus-grade value, or paying the cap premium out of habit?"
  -> Top sessions still surfaced, but with cap-% framing where useful
  -> Footnote: "For live cap %, run /usage in Claude Code"
       |
[3: User has aha moment that doesn't depend on dollar amounts]
```

### Configuration system

`~/.mileage/config.json`:
```json
{
  "version": 1,
  "plan": "max-100",
  "preferences": {
    "currency": "USD",
    "show_dollars_anyway": false,
    "waste_threshold_usd": 5
  }
}
```

Plans enum:
- `api` — pay-per-token, dollar framing
- `pro` — Claude Pro
- `max-100` — Claude Max 5×
- `max-200` — Claude Max 20×
- `cursor-pro`, `copilot` — placeholder, defaults to token framing
- `unknown` — show everything

CLI:
- `mileage config get` — dump config
- `mileage config set-plan <plan>` — declare plan
- `mileage config set-currency <code>` — for non-USD users
- First-run prompt in `mileage show` if not configured

### Rate-limit hit detection

Parse JSONL for entries indicating the user got rate-limited. Specific patterns to detect:

1. Tool result `is_error: true` with content containing `"rate_limit_error"` or HTTP 429
2. Assistant entries with `stop_reason: "error"` and rate-limit indicators
3. Specific Claude Code error message patterns (e.g., "Your account has been rate limited")

Store in DB as a new `rate_limit_hits` table:
```sql
CREATE TABLE IF NOT EXISTS rate_limit_hits (
  id          TEXT PRIMARY KEY,
  timestamp   INTEGER NOT NULL,
  session_id  TEXT,
  window      TEXT,  -- '5h' | '7d' | 'unknown'
  raw_message TEXT,
  created_at  INTEGER NOT NULL
);
```

Surface in `mileage show`:
- 0 hits this week: `Rate-limit hits   0 this week ✓`
- N hits this week: `Rate-limit hits   ⚠ 3 this week (Tue 4pm, Sat 11am, Sun 3pm)`

This is **ground-truth exact data** — no heuristics, no community-approximated caps. Just count what's already in the JSONL.

### stats-cache.json cross-check

Read `~/.claude/stats-cache.json` if present. Provides per-day, per-model token totals (last computed at most every few days). Use to:

1. **Cross-check our own JSONL parsing** — log a warning when our token totals diverge from stats-cache by >10%
2. **Backfill recent days** if we missed JSONL files (rare but possible)
3. **Per-model rollup** — supplements our own per-session model tracking

Implementation: read on `mileage sync`, store as a side-table or merge into existing snapshots.

### Tier-flex audit

The killer Max-plan aha feature. Pure SQL over existing data:

```sql
SELECT
  model_id,
  COUNT(*) AS sessions,
  SUM(CASE WHEN session_id IN (SELECT session_id FROM attributions) THEN 1 ELSE 0 END) AS sessions_with_outcome,
  CAST(SUM(CASE WHEN session_id IN (SELECT session_id FROM attributions) THEN 1 ELSE 0 END) AS REAL)
    / NULLIF(COUNT(*), 0) AS yield_rate,
  AVG(cost_usd) AS avg_cost,
  SUM(cost_usd) AS total_cost
FROM events
WHERE type = 'session'
  AND timestamp >= ?
GROUP BY model_id
ORDER BY total_cost DESC
```

Surface in `mileage show` when 2+ models have meaningful data (≥5 sessions each):

```
Tier-flex audit (this month):
  claude-opus-4-7    47 sessions   18 with outcomes (38%)   avg $46/session
  claude-sonnet-4-6  23 sessions   16 with outcomes (70%)   avg $9/session
  
  ⚠ Your Opus sessions are ~5× more expensive but yield outcomes less often.
    Try Sonnet first; reserve Opus for sessions where Sonnet stalls.
```

The recommendation logic:
- If Opus yield-rate < Sonnet yield-rate AND avg-cost-ratio > 3: surface tier-flex warning
- If Sonnet yield-rate < Haiku yield-rate AND user has both: similar warning
- Otherwise: silent (don't nag when ratios are reasonable)

### Plan-aware display

Single `renderLast7Days` function that branches on `config.plan`:

| Plan | Headline metric | Secondary | Tertiary |
|---|---|---|---|
| `api` | Spend $X | Cost-per-Ship $Y/commit | Tokens, sessions |
| `max-100`, `max-200`, `pro` | Tokens used (vs your typical week) | Rate-limit hits | Cost-equivalent (informational, dim) |
| `unknown` | Both dollar and token framings | All of the above | All of the above |

For Max users specifically, the "cap %" framing is **NEVER computed against an approximated absolute cap**. Instead:
- "12.4M tokens this week (180% of your 30-day median)"
- "Rate-limit hits: 0 this week ✓"
- "For real cap %, run `/usage` in Claude Code"

This is honest. We don't have the cap number; we have the user's own baseline. That's enough.

## Phase 2b — Sticky behavioral features

### Self-tag system (`mileage tag`)

Interactive walk through recent unscored sessions. Triggered by `mileage tag` or auto-prompted at the end of `mileage sync` when ≥3 unscored sessions exist.

```
$ mileage tag

Session 1 of 3:
  Sat 14:32 · 2h45m · Opus · $94 · 0 commits

  What happened?
    [s] shipped
    [e] exploring
    [d] debugging
    [x] dead-end
    [k] skip
    [q] quit

> _
```

Storage: `session_tags` table:
```sql
CREATE TABLE IF NOT EXISTS session_tags (
  session_id  TEXT PRIMARY KEY,
  tag         TEXT NOT NULL CHECK (tag IN ('shipped','exploring','debugging','dead-end')),
  tagged_at   INTEGER NOT NULL
);
```

Tags influence display:
- `shipped` without attribution → still counts as outcome (primary signal beats automation)
- `exploring` → excluded from cost-per-ship denominator (research ≠ waste)
- `dead-end` → counted in waste detection regardless of cost threshold
- `debugging` → neutral, just informational

### Git post-commit hook

`mileage install-hook` writes `.git/hooks/post-commit`:
```sh
#!/bin/sh
mileage sync --silent --since 24h &
```

Plus High-tier attribution: hook checks for recent JSONL writes (within last 5 min) on Mileage-tracked projects, writes `(session_id, commit_hash, tier=high, conf=0.75)` to `attributions`.

`mileage uninstall-hook` reverses cleanly.

Bonus: hook can optionally prompt for self-tag inline:
```
$ git commit -m "fix auth race condition"
[mileage] You spent $84 in this Claude session.
  What happened? [s]hipped / [d]ead-end / skip:
> s
```

Configurable; off by default. The forcing-function design.

### Behavioral patterns

Pattern detection over rolling 30 days. Surface findings in `mileage show` when statistically meaningful (≥3 occurrences of a pattern).

Patterns to detect:
- **Time-of-day:** "Your morning sessions yield 1.8 commits avg; afternoon yield 0.3"
- **Day-of-week:** "Your Saturday sessions cost 3× your Tuesday sessions for similar output"
- **Model × outcome:** "Long Opus sessions (>1h) with 0 commits: 8 sessions, $642 total"
- **Project:** "Project X has 5× your typical cost-per-ship — investigate?"

Implementation: `src/compute/patterns.ts` runs a small library of detectors, each returning `{pattern, severity, recommendation}` if triggered. Render layer shows top 1-2 by severity.

### Code Survival Rate (7d)

Track which AI-attributed lines are alive 7 days later. Pulled forward from V0.3 because Max-plan users need quality signals.

For each commit in `events`:
1. On commit ingestion: record the `(commit_hash, file_path, lines_added)` tuples
2. Background job (runs on `mileage sync`): for commits ≥7d old, check `git log --follow --numstat` for revisions to the same lines
3. Compute `survival_rate_7d = lines_still_present_at_7d / total_lines_added`

Schema:
```sql
CREATE TABLE IF NOT EXISTS commit_survival (
  commit_hash      TEXT NOT NULL,
  evaluated_at     INTEGER NOT NULL,
  lines_added      INTEGER NOT NULL,
  lines_surviving  INTEGER NOT NULL,
  window_days      INTEGER NOT NULL,
  PRIMARY KEY (commit_hash, window_days)
);
```

Surface in `mileage show`:
```
Code health:
  This week's commits: 87% still alive at 7d (vs 92% your baseline)
  Most-churned files: auth.ts (3 revisions), api/users.ts (2 revisions)
```

Note: file-level only. Line-level requires `git blame` which is privacy-line-crossing. Track which *files* got re-edited within 7 days.

## Phase 2c — Visual polish + brand artifact

### Heatmap (`mileage show --heatmap`)

GitHub-contribution-graph style, 90 days. Terminal-rendered with ANSI 256-color (truecolor where supported).

Layout:
```
                    May
        Mo Tu We Th Fr Sa Su
  Week 1 ░  ▒  ▓  █  ▒  ░  .
  Week 2 ▒  ▓  █  ▒  ░  .  .
  ...
```

Color encoding:
- Hue: green (efficient) → yellow (neutral) → red (waste)
- Brightness: dim (low spend) → bright (high spend)
- Empty cell: `.` for "no AI sessions this day"

For Max users, color encodes "your-typical-day baseline" rather than absolute cost. For API users, color encodes dollar Cost-per-Ship.

No streaks — research warned they're anxiety-inducing.

### Sparklines + bar charts

Inline charts in `mileage show` and elsewhere:

```
Daily spend last 30 days:  ▁▁▂▃▂▅▇▆▅▃▂▁▂▄▆▇▅▃▂▁▁▂▃▅▇▆▅▃▂▁    $98 avg

By model this month:
  Opus 4.7    ████████████████████  $1,872 (65%)
  Sonnet 4.6  ████                    $487 (17%)
  Haiku 4.5   ▏                        $12 (<1%)
```

All using Unicode block characters; degrades cleanly when terminal can't render.

### Weekly report (`mileage report --week`)

Copy-pasteable markdown for sharing. The brand-building artifact for public dogfooding.

```markdown
# My Mileage Week — May 17-23, 2026

**Tokens used:** 12.4M  (180% of my 30-day median)
**Top model:** Claude Opus 4.7 (73% of usage)
**Rate-limit hits:** 0 ✓

## Surprising findings
- A 3-hour Wednesday debug session cost ~14% of my 5-hour cap and shipped nothing.
- Sonnet sessions averaged 1.7 commits/session; Opus averaged 0.4. I'm flexing Opus out of habit.

[mileage 0.2 · github.com/mileage-cli/mileage]
```

Output is plain markdown, copy-paste friendly. User dogfoods this format publicly every week → standards-play seed + personal-brand artifact.

### `mileage review` (interactive forcing function)

Separate from `mileage report --week` (which is a copy-paste artifact). `mileage review` is an interactive guided walk:

```
$ mileage review

Reviewing your top 5 expensive sessions this week:

1. Sat 14:32 · 2h45m · Opus · 14% of cap · 0 commits
   Was this worth it?
     [y] yes, even though it didn't ship
     [n] no — what should I have done differently?
     [s] skip

> n

   What pattern is this? (your answers train future suggestions)
     [1] Wrong tool — Sonnet would have worked
     [2] Bad scoping — should have planned more first
     [3] Stuck on something hard — should have walked away sooner
     [4] Exploring (excluded from waste metrics)
     [5] Other / describe

> 3

...
```

Tags accumulate; over time the tool learns the user's patterns and starts surfacing them proactively. Closes the perception/reality loop METR documented.

## Edge cases

- **First run with no config:** prompt for plan; defaults to `unknown` if user skips
- **Plan change mid-tracking:** historical data displays in the framing of the historical plan if logged; future data uses new plan
- **No JSONL files yet:** existing empty-state message in V0.1 still applies
- **stats-cache.json missing or corrupt:** skip cross-check silently
- **Rate-limit detection false positives:** be conservative on patterns; false positive means we falsely show "you got rate-limited" which is worse than missing one
- **Self-tag conflict with auto-attribution:** user tag wins; auto-proxies become advisory only

## Out of scope (V0.3+)

- LLM-as-judge intent fulfillment
- Cognitive complexity / Semgrep / jscpd diff scanning (Quality Mode opt-in)
- YPT formula upgrade to log-normal CDF
- Cross-tool ingestion (Cursor, Copilot)
- Anthropic Admin API integration for org-admin API users
- Multi-user / cloud sync / web UI
